# PayOps Console

一个面向商户运营、财务、风控、审计角色的支付运营后台作品集项目。

项目重点不是接真实支付网关，而是用前端方式把支付中后台最核心的业务链路做完整、做清楚、做得可演示。

## 项目定位

`PayOps Console` 主要用于展示以下能力：

- 支付中后台的信息架构设计
- 多角色登录与 RBAC 权限控制
- 交易、退款、对账、风控、审计等业务模块拆分
- 列表页、详情页、处理弹窗、状态流转等典型后台交互
- 基于 React + TypeScript 的工程化组织方式
- 基础单测与核心业务冒烟测试能力

当前实现以本地 mock 数据和 Provider 状态管理为主，优先保证页面演示效果、链路完整性和代码可维护性。

## 当前能力

目前项目已经具备以下可演示能力：

- 登录页、演示账号快速登录、登录失败提示
- 基于 `用户 -> 角色 -> 权限` 的 RBAC 解析、登录态持久化、路由守卫、菜单级权限控制、403/404 页面
- 基于当前用户权限计算默认首页，避免误跳到无权限页面
- 路由懒加载与统一 Suspense 加载占位
- 图表看板入口
- 交易列表、筛选、排序、分页、订单详情、订单时间线
- 从订单详情发起退款，并在退款管理页完成状态流转
- 对账中心批次汇总、差异单筛选与处理闭环
- 风险告警列表、详情抽屉与处理闭环
- 审计日志查询、筛选、详情查看，并接入登录、退出、退款、对账、风控等关键留痕
- 访问控制页：用户管理、角色管理、权限字典
- 支持创建用户、编辑用户、启用/停用账号、复制角色、编辑角色权限、删除未绑定自定义角色
- 覆盖登录回跳、权限拦截、退款 -> 对账 -> 风控 -> 审计主链路，以及访问控制主链路的测试

## 快速开始

### 环境要求

- Node.js `v20.20.2`
- npm

仓库内提供了 `.nvmrc`，推荐直接使用 `nvm` 切换版本。

### 安装依赖

```bash
nvm use
npm install
```

### 启动开发环境

```bash
nvm use
npm run dev
```

默认访问地址：

```bash
http://localhost:5173
```

### 生产构建


```bash

npm run test:run
```

## 业务链路

当前最完整、最适合演示的主链路如下：

1. 使用管理员或运营账号登录系统
2. 在交易列表进入订单详情
3. 从订单详情发起退款
4. 在退款管理页查看退款状态或完成审核
5. 进入对账中心处理差异单
6. 进入风险告警页完成告警处置
7. 最后在审计日志页查看关键动作留痕

## 项目结构

```text
src/
├── app/                 # 应用壳、Provider、路由
├── features/            # 业务模块
│   ├── auth/            # 登录、会话、演示账号
│   ├── access-control/  # 用户、角色、权限字典与 RBAC Provider
│   ├── dashboard/       # 图表看板
│   ├── transactions/    # 交易列表、订单详情
│   ├── refunds/         # 退款管理与退款状态流转
│   ├── reconciliation/  # 对账中心
│   ├── risk-alerts/     # 风险告警
│   └── audit-logs/      # 审计日志
├── shared/              # 常量、通用 UI
├── styles/              # 全局样式
└── tests/               # 冒烟测试与测试初始化
```

- 登录页与默认首页跳转逻辑

- 交易详情 -> 退款 -> 对账 -> 风控 -> 审计主链路

- 关键业务工具函数和主要页面行为

如果你想快速验证核心链路，优先执行：

```bash
npm run test:run -- src/tests/smoke-flow.test.tsx
```

## 已知限制

- 目前没有接入真实后端接口，主要依赖本地 mock 数据与 Provider 管理状态
- 当前更强调业务演示和作品集展示，不强调生产级安全、接口容错和部署体系
- 退款、对账、风控、审计、访问控制等运行时数据仍以内存态为主，尚未统一做本地持久化和重置演示数据机制
- 生产构建仍存在一个 Vite chunk size warning，但不影响本地运行和演示

## 接下来可以怎么继续

如果你想基于这个仓库继续迭代，当前最值得投入的方向是：

- 为运行时 Provider 增加本地持久化和“重置演示数据”
- 统一页面文案、空状态和反馈文案
- 继续收拢重复的筛选区、统计卡片和表格模式
- 优化打包体积，按路由或模块拆包
- 在不增加太多维护成本的前提下补少量高价值测试
- 将当前账号会话和 mock Provider 逐步替换为真实接口
